Architecture

Designed Like a Real Database Engine

Clean, layered architecture with separation of concerns — from TCP socket handling to persistence layer.

01

TCP Listener

Accepts multiple concurrent client connections via ServerSocket. Thread-per-client model with ExecutorService thread pool.

02

RESP2 Protocol Parser

Hand-written binary-safe parser and encoder. Supports Simple Strings, Errors, Integers, Bulk Strings, and Arrays.

03

Command Router

Dispatches parsed commands to the appropriate handler — Strings, Lists, Sets, Hashes, Keys, PubSub, Transactions, Server.

04

Data Store Layer

ConcurrentHashMap backing store with ExpiryManager running background cleanup and lazy deletion on access.

05

Persistence Layer

Dual persistence with AOF (write-ahead log) for durability and RDB (JSON snapshots) for fast point-in-time recovery.

Why Thread-per-Client?

Simpler to understand, debug, and explain. Each client gets a dedicated thread from an ExecutorService thread pool. The data store itself is thread-safe via ConcurrentHashMap.

Why Dual Persistence?

AOF = durability — every write is logged. RDB = performance — periodic snapshots are fast to load. On startup: load RDB first, then replay AOF for writes since last snapshot.

Why RESP2?

Simple enough to implement from scratch, and gives free compatibility with redis-cli, Jedis, Lettuce, and every Redis client library. It's not a toy — it speaks a real protocol.
